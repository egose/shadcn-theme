# AI-Friendly Theme Package

Use this skill for the two publishable theme packages in this workspace:

- `packages/react` → published as `@egose/shadcn-theme`
- `packages/angular` → published as `@egose/shadcn-theme-ng` (and the `-tw` tailwind-prefix variant)

The goal is not "good docs inside this repo." The goal is: after a consumer installs the package, their editor and AI assistant can discover the public surface, understand the import forms, and produce correct usage.

## Critical Mental Model: Source `package.json` Is Not the Published Package

Both `packages/react/package.json` and `packages/angular/package.json` are marked `"private": true` and use `"version": "0.0.0-PLACEHOLDER"`. They are NOT what npm publishes. The publishable artifact is built into `packages/<context>/dist/` and its `dist/package.json` is generated by `scripts/publish.mjs` at release time.

When evaluating installed-consumer discoverability, inspect `dist/`, not the source `package.json`. A "fix" that only edits the private source `package.json` will not change what consumers see.

What the installed consumer actually sees:

1. `dist/package.json` — generated by the publish script
2. shipped declaration files under `dist/` (React: `*.d.ts` / `*.d.mts`; Angular: `<lib>/index.d.ts`)
3. shipped JavaScript entrypoints under `dist/` (React: `*.mjs` (ESM) + `*.js` (CJS); Angular: `<lib>/fesm2022/<lib>.mjs`)
4. `dist/README.md` — copied from the package root by the publish script
5. `LICENSE` — copied alongside

## Success Criteria

An installed consumer should be able to answer these without opening repo source files:

1. How do I import a component from this package? (React deep import vs Angular subpath)
2. What components, hooks, utils, and layouts are public?
3. Which peer dependencies or runtime assumptions do I need?
4. What is the shortest correct happy-path example for my framework?
5. For Angular: do I need the plain package or the `-tw` tailwind-prefixed variant?

## Workspace Shape

### Root

- monorepo is driven by `pnpm` plus per-package `pnpm-lock.yaml` files
- `Makefile` orchestrates `make install` (installs root, `packages/react`, `packages/angular`, and both `@examples/*`), `make nextjs` (runs the React Next.js example), and `make detect-secret`
- there is no test runner at the root; the root `test` script intentionally exits non-zero
- `scripts/publish.mjs --context react|angular` performs the actual npm release

### `packages/react` → `@egose/shadcn-theme`

- source directories are the tsup `entry`: `components/`, `hooks/`, `utils/`, `layouts/`
  - `components/ui/<name>.tsx` — the bulk of the UI primitives (button, dialog, select, …)
  - `components/form/<name>.tsx` — `react-hook-form`-backed form fields
  - `components/widgets/<name>/` — compound widgets such as `dialog-manager`
  - `hooks/use-*.tsx` — `use-clipboard`, `use-mobile`
  - `utils/<name>.ts` — `date`, `time`, `ui` (the `cn` helper lives here)
  - `layouts/<name>/` — `simple`, `sidebar1`
- scripts: `pnpm dev` (`tsc -w`) for source watching; `pnpm bundle` (`tsup`) for the publishable artifact. There is NO `pnpm build` script in the package — `build` is reserved for the example app under `@examples/nextjs` (where `pnpm build` runs `next build`). Do not add a `build` script here without an explicit reason; it muddies the package-vs-example distinction.
- `tsup.config.ts` emits CJS + ESM with `splitting`, `dts: true`, `external: ['react', 'react-dom', 'react-hook-form', 'sonner']`
- `postbundle` runs `prepend-use-client.mjs`, which strips any existing `"use client";` from `dist/**/*.cjs` and `dist/**/*.js` and prepends a single fresh `"use client";` directive at the top of every emitted file. Note: tsup actually outputs `*.js` (CJS) + `*.mjs` (ESM) for this repo — there are no `.cjs` files, so the `.cjs` glob is dead and only the `.js` glob matches. The `.mjs` files get their `"use client";` from the in-process `prependUseClient` esbuild plugin in `tsup.config.ts`, which transforms source via `build.onLoad` before emit. This is required for React Server Components compatibility and must keep working after any build change.
- `peerDependencies`: `react`, `react-dom`, `react-hook-form`, `sonner`. These match the `tsup` `external` list and must stay in sync.
- consumer examples live in `packages/react/@examples/nextjs/` — that example directory exposes its own `pnpm build` (`next build`), `pnpm dev` (`next dev`), `pnpm start`, `pnpm lint`. The package directory itself only has `pnpm dev` and `pnpm bundle`; `pnpm --dir packages/react build` will fail with "No build script" by design.

### `packages/angular` → `@egose/shadcn-theme-ng`

- one Angular CLI workspace, one library per component under `projects/<name>/`
  - each lib has `projects/<name>/ng-package.json` with `lib.entryFile = src/public-api.ts`
  - `public-api.ts` re-exports the public surface from `src/lib/*`
  - each lib builds via `@angular/build:ng-packagr` into `dist/<name>/`
  - emitted shape per lib: `index.d.ts`, `fesm2022/<name>.mjs`, plus `*.d.ts` for sub-files
- `build-all.mjs` is the canonical bundler:
  - reads `tsconfig.json` `paths` for keys starting with `@egose/shadcn-theme-ng`
  - runs `pnpm ng build <name>` for each matching project that exists under `projects/`
  - writes `dist/exports.json` mapping `./<name>` → `{ types, default }` for every successfully built lib
  - rewrites package imports inside emitted `.mjs` files from `@egose/shadcn-theme-ng/...` → `<newPkgName>/...`, where `newPkgName` is `@egose/shadcn-theme-ng` for the bare build or `@egose/shadcn-theme-ng-<bundle>` for a bundle variant
  - rewrites tailwind class-prefix tokens in `.mjs` files according to `twPrefix = bundle ? `${bundle}:` : ''`:
    - **bare build** (no `bundle` argv): `tw:` → `''`, i.e. the `tw:` prefix is STRIPPED from every class string. The bare `@egose/shadcn-theme-ng` consumer must use plain (unprefixed) tailwind.
    - **`-tw` variant** (`bundle = 'tw'`): `tw:` → `tw:` (identity no-op), so the prefix is KEPT. The `-tw` consumer must configure tailwind with the `tw:` prefix.
- `packages/angular/package.json` declares `"bundles": ["tw"]`, which `scripts/publish.mjs` iterates to also publish `@egose/shadcn-theme-ng-tw`
- root `angular.json` enumerates every sub-project; keep it in sync when adding or removing a component library
- consumer example lives in `packages/angular/@examples/standard/` — that example directory exposes its own `pnpm build` (`ng build`), `pnpm start` (`ng serve`), `pnpm watch`, `pnpm test`. The package directory itself only has `pnpm dev` (`ng serve`), `pnpm watch`, `pnpm test`, `pnpm bundle`; `pnpm --dir packages/angular build` is intentionally absent. `build-all.mjs` runs `pnpm ng build <name>` per-sub-library (an explicit `ng` invocation, NOT `pnpm build`), so removing the package-level `build` script does not affect bundling.

### Release flow

`scripts/publish.mjs --context <react|angular>`:

1. reads `packages/<context>/package.json` (with the `0.0.0-PLACEHOLDER` version and `"PLACEHOLDER"` for `license`/`author`)
2. reads the ROOT `package.json` and copies the real `license` and `author` values into the per-package data (overriding the `"PLACEHOLDER"` strings) so both published packages stay in sync with the monorepo's license/authorship without duplicating those values per-package
3. resolves the target version via `npm show <name> version` plus `--major|--minor|--patch|--version`
4. for each bundle in `['', ...packageData.bundles]`:
   - runs `cd packages/<context> && pnpm bundle <bundle>`
   - replaces `0.0.0-PLACEHOLDER` in `version`, `dependencies`, `peerDependencies`
   - picks a curated allow-list of fields via `_.pick`: `version`, `description`, `keywords`, `homepage`, `bugs`, `license`, `author`, `sideEffects`, `files`, `repository`, `dependencies`, `peerDependencies`, `release`, `engines`, `main`, `module`, `types`, `exports`
   - copies `LICENSE` (from the repo ROOT — the single canonical Apache-2.0 LICENSE file), `README.md` (per-package), and `llms.txt` (per-package, if present) into `dist/`
   - for Angular: sets `exports` from `dist/exports.json`
   - writes `dist/package.json` and runs `npm publish --access public`

Consequence: any field not in the publish script's allow-list will NOT reach the installed consumer even if you add it to the source `package.json`. To ship a new field you must extend `scripts/publish.mjs`'s `_.pick` allow-list (currently: `version`, `description`, `keywords`, `homepage`, `bugs`, `license`, `author`, `sideEffects`, `files`, `repository`, `dependencies`, `peerDependencies`, `release`, `engines`, `main`, `module`, `types`, `exports`). The per-package `license` and `author` fields hold `"PLACEHOLDER"` and are ONLY meaningful after the publish script substitutes them from the root `package.json` — do NOT edit them per-package; change the root `package.json` instead. The single `LICENSE` file at the repo root (Apache-2.0) is copied into both `dist/` directories at publish time; per-package `LICENSE` files are NOT required. After building Angular, `scripts/publish.mjs` also deletes `dist/exports.json` — a build-internal artifact used only to seed `dist/package.json`'s `exports` field — to prevent it from leaking onto the npm registry.

## Priority Order

Work in this order:

1. Generated `dist/package.json` (especially `exports`, `types`, `main`/`module`, `sideEffects`, `peerDependencies`)
2. `dist/exports.json` (Angular) — generated by `build-all.mjs`; must list every existing sub-project
3. Shipped declaration files under `dist/` — what editors actually read after install
4. `dist/README.md` — copied verbatim from the package root; must be self-sufficient for the happy path
5. JSDoc on the highest-value exported APIs (so it survives onto the generated declarations)
6. `tsup.config.ts` / `build-all.mjs` / `angular.json` consistency (entrypoints vs published `exports`)
7. Optional `llms.txt`
8. `@examples/*` consumers only as a consistency check

Do not treat `llms.txt` or `@examples/*` as a substitute for correct metadata, types, and README examples.

### 2. Verify the generated publish surface

Inspect `dist/` after a real build. Do not trust the source `package.json` alone.

- `dist/package.json` exists and has the real version (no `0.0.0-PLACEHOLDER`)
- for Angular: `dist/exports.json` is present and matches the list of `projects/<name>/` directories and the `paths` in `tsconfig.json`
- for Angular: `dist/package.json` `exports` was populated from `exports.json` (the publish script does this; if you bypass publish, `dist/package.json` lacks `exports`)
- for React: confirm `exports` is declared. The source `packages/react/package.json` now ships an `exports` map covering `./components/ui/*`, `./components/form/*`, `./components/widgets/dialog-manager` (+ subfiles), `./hooks/*`, `./utils/*`, `./layouts/simple` (+ subfiles), `./layouts/sidebar1` (+ subfiles), and `./package.json`. Verify the published `dist/package.json` carries it through (the `scripts/publish.mjs` allow-list includes `exports`). Maintain the wildcards every time new top-level subpaths or layout/widget subdirs are added.
- `peerDependencies` on the published package match the runtime assumptions (React: `react`, `react-dom`, `react-hook-form`, `sonner`)
- `sideEffects: false` is set on both source `package.json` files and carried through to `dist/package.json`. The `"use client";` directive is not a side effect, so this is generally safe for tree-shaking, but verify no shipped module performs top-level registration (DOM listeners, `valtio` proxy snapshots, etc.) before changing the value.
- any field you intend to ship is in the `scripts/publish.mjs` allow-list, otherwise it silently drops at publish time

Verify with a real build when practical: `pnpm --dir packages/react bundle` or `pnpm --dir packages/angular bundle` (NOT `pnpm build` — that script is reserved for the `@examples/*` apps and is intentionally absent from the package directories). Then inspect `dist/`. Use `npm pack --dry-run` inside `dist/` to see exactly what the installed consumer receives.

### 3. Verify the React `"use client"` step

This is the single most failure-prone part of the React build. Two mechanisms cooperate:

- the `prependUseClient` esbuild plugin inside `tsup.config.ts` runs `build.onLoad` on every `.(ts|tsx|js|jsx)$` SOURCE file and prepends `"use client";` if absent. Because this transforms source before emit, BOTH output formats inherit the directive — i.e. `dist/**/*.mjs` already carry `"use client";` from the source pass.
- the `postbundle` script `prepend-use-client.mjs` then runs over `dist/**/*.cjs` and `dist/**/*.js`, strips ALL existing `"use client";` occurrences, and prepends a single fresh one. Because tsup's CJS output extension is `.js` here (not `.cjs`), only the `.js` glob actually matches anything; the `.cjs` glob is currently dead code.

Verify:

- every `dist/**/*.js` (CJS) file starts with exactly one `"use client";` line, with no duplicates or mid-file occurrences
- every `dist/**/*.mjs` (ESM) file also starts with `"use client";` (delivered by the tsup source-pass plugin, NOT by `prepend-use-client.mjs`)
- if you change `tsup.config.ts` `format` (e.g. switch CJS extension to `.cjs` via `--out-extension .cjs`), update `prepend-use-client.mjs`'s glob accordingly, or the directive will be missing from CJS output
- if you add a new `entry` to `tsup.config.ts`, both mechanisms still apply (the plugin keys off file extension, not entry name)

### 4. Verify the Angular subpath export map

Angular consumers import one component at a time via subpaths. The map has to be complete and accurate.

- every directory under `projects/` appears in `tsconfig.json` `paths` as `@egose/shadcn-theme-ng/<name>`
- every such path has a matching `projects/<name>/` directory with a valid `ng-package.json` and `src/public-api.ts`
- `angular.json` enumerates the same set of projects
- after `build-all.mjs`, every project that built successfully has an entry in `dist/exports.json`, and the entry's `types` and `default` point at files that actually exist in `dist/<name>/`
- the `tw` bundle pass rewrites package-import refs to `@egose/shadcn-theme-ng-tw/...` in `.mjs` files and keeps the `tw:` class-string prefix intact; the bare build rewrites those same refs to `@egose/shadcn-theme-ng/...` and STRIPS the `tw:` prefix entirely. Verify a single variant's `.mjs` does not leak cross-variant refs (e.g. `-tw` output referencing `@egose/shadcn-theme-ng/` without the suffix, or vice versa).

Common drift signs:

- a new project directory under `projects/` that is missing from `tsconfig.json` `paths` or `angular.json` — it will silently not build. (`angular.json` uses the same set as `projects/`; both must be updated when adding a component.)
- a `paths` entry with no matching project — `build-all.mjs`'s `if (!projects.includes(name)) return` skips it silently. (Historically this repo had 18 phantom `paths` entries for `alert-dialog`, `card`, `command`, `hover-card`, `pagination`, `progress`, `scroll-area`, `skeleton`, `slider`, `table`, `toggle`, `toggle-group`, `typography`, etc. None had matching `projects/<name>/` dirs and all were silently skipped every build; they have been cleaned up — keep an eye out for regressions.)
- a stale `dist/exports.json` from a previous build listing a removed component

### 5. Verify shipped declaration files

Prioritize what TypeScript and the editor read after install.

- React: every entry from `tsup.config.ts` (`components`, `hooks`, `utils`, `layouts`) emits per-file `*.d.ts` and `*.d.mts` in `dist/`. The React package is fully flat — there are NO barrel `index.ts` files anywhere (`packages/react/index.ts`, `components/ui/index.ts`, `hooks/index.ts`, `utils/index.ts` all absent). Each component/hook/util/layout is its own deep-import target.
- React: deep declarations such as `dist/components/ui/button.d.ts` and `dist/components/ui/button.d.mts` exist and expose the intended public types
- Angular: `dist/<name>/index.d.ts` exposes the `public-api.ts` surface cleanly
- important types (component props, form field option types, layout context types) are reachable from the published declarations without forcing deep `src/*` imports
- exported API shapes are specific enough to drive good editor hover and autocomplete
- key JSDoc survives onto the generated declarations

Do not assume source comments or source exports are enough. Inspect the emitted `.d.ts`/`.d.mts` files.

### 6. Verify the public TypeScript surface

- React has NO barrel `index.ts` files; the public surface is the set of deep-import paths. If you want a single-import experience you would have to ADD a barrel — but only do so if `tsup.config.ts` `entry` and any new `exports` map reflect it. Default to documenting deep imports.
- public APIs avoid `any` and overly loose structural types unless there is a real need
- default vs named exports are deliberate and consistent (this repo is named-export-first for React UI primitives)
- docs and examples do not rely on `src/*` or `dist/*` deep imports that the publish surface does not actually expose
- Angular `public-api.ts` files export exactly the components, directives, pipes, and types a consumer needs

### 7. Verify README quality for installed consumers

Both `packages/react/README.md` and `packages/angular/README.md` are copied verbatim into `dist/` by the publish script. The current React README is only a color-palette table, and the Angular README is a single `pnpm ng generate` note — both are inadequate for an installed consumer.

Prefer the shortest correct example that matches the actual public API.

For React:

- show install: `npm install @egose/shadcn-theme react react-dom react-hook-form sonner`
- list required peer dependencies
- show the canonical deep import for one UI primitive, one form field, one hook, one util, and one layout
- mention the `"use client"` boundary so Next.js / RSC consumers know all shipped modules are client components

For Angular:

- show install: `npm install @egose/shadcn-theme-ng` for plain tailwind, OR `@egose/shadcn-theme-ng-tw` for apps configured with a `tw:` tailwind prefix
- mention required peer Angular version and `@spartan-ng/brain`
- show the subpath import form: `import { HlmButtonModule, HlmButton } from '@egose/shadcn-theme-ng/button'` (and the equivalent `-tw/...` form for the prefixed variant)
- explicitly call out the prefix semantic: the bare package ships class strings WITHOUT a tailwind prefix (plain tailwind only); the `-tw` package ships class strings WITH the `tw:` prefix (consumer tailwind must be configured with `tw:` prefix)
- point at `packages/angular/@examples/standard` as the working consumer example — its `src/app/...` files use `@egose/shadcn-theme-ng/button`, `.../form-text-input`, `.../layout-simple`, etc.

Keep README examples package-boundary-safe and copy-pasteable. Do not reference `src/*` or internal file paths.

### 8. Add JSDoc only where it helps the most

Prefer JSDoc on:

- the `cn` helper in `utils/ui.ts` and other utility entry points
- top-level exported components with non-obvious props (form field wrappers, dialog-manager, layouts)
- Angular component/directive classes that consumers configure via inputs
- shared option/variant types driving `class-variance-authority`

Confirm the JSDoc lands on the generated `.d.ts`/`.d.mts` that the editor actually shows. Key JSDoc has been added to the highest-value React exports: `cn`, `useClipboard`, `useIsMobile`, `convertToHours`, `isEqualDate`/`isEqualDates`, `buttonVariants` + `Button` + `ButtonProps` + `VariantType`/`SizeType`/`VariantStyleType`, `FormTextInput` + `FormTextInputProps`, `HookFormRules`, `DialogProvider` + `useDialog` + `createTypedDialog` + `DialogProps` + `TypedDialogComponent`, `SimpleLayout` + `LayoutProps` + `MenuItem` + `UserMenuSection`, `SidebarLayout` + `headerStore` + `setLayoutHeader` + `ISidebarData` + `ISidebarContext` + `INavUser` + `INavUserMenuItem`. Verify each rebuild still emits the JSDoc onto the corresponding `.d.mts` file. Keep JSDoc free of JSX syntax (the tsup `prependUseClient` plugin's `build.onLoad` parses `.tsx` files and chokes on JSX inside JSDoc — use plain text like `-- args.title rendered here --` instead of `{/* ... */}`).

Angular JSDoc is still a candidate; the Angular public surface is mostly Angular decorators + class-based components, and `public-api.ts` already exports everything, so the marginal value is lower than the React work. Skip unless prompted.

### 9. Check `@examples/*` only for drift

Use `packages/angular/@examples/standard` and `packages/react/@examples/nextjs` to detect mismatches with the published surface, not as the primary source of truth.

- The Angular `@examples/standard` example IS a real consumer: it imports `@egose/shadcn-theme-ng/<name>` subpaths throughout `src/app/...` (button, label, sheet, autocomplete, alert, form-text-input, layout-simple, …). Treat it as the canonical Angular consumer pattern.
- The React `@examples/nextjs` example is NOT a consumer of the published `@egose/shadcn-theme` package. A repo-wide grep finds no `@egose/shadcn-theme` import in `packages/react/@examples/nextjs/**` — the only "egose" hits are string literals like `"egose-inc"`. The example appears to maintain its own copies of components under its own `components/`. Do not cite it as evidence of how `@egose/shadcn-theme` is consumed; flag the gap instead.

### 10. Decide whether `llms.txt` is justified

A `llms.txt` index has been added to the React package at `packages/react/llms.txt`. It covers:

- install + peer dependencies
- the full subpath map (ui primitives, form fields, widgets, hooks, utils, layouts)
- canonical usage patterns (a Next.js-style client component demo and a promise-based dialog-manager flow)
- notable conventions (`"use client"` boundary, `Button` variants, `react-hook-form` integration through `inputProps`/`HookFormRules`, the required `aslink` layout prop)

`scripts/publish.mjs` now copies `llms.txt` into `dist/` alongside `README.md` and `LICENSE`, and the React `package.json`'s `files` field lists `llms.txt` so `npm pack` ships it. Keep the index short and index-like — update it whenever the public subpath surface changes (new top-level subpath, new layout/widget subdir, etc.).

The Angular package does not currently have an `llms.txt`. Recommend adding one only after the broken-component issues (see Common Findings 5) are resolved and the actual public surface matches the README's `Available subpaths` section.

## Common Findings In This Repo

Prioritize these findings in review output:

1. React `exports` map drifts from the actual `dist/` subpath structure (now declared in source `package.json`; verify the wildcards still match every time a new top-level subpath or layout/widget subdir is added)
2. React `dist/README.md` does not teach install, peers, or import forms (the README has been rewritten; keep it in sync with the public surface)
3. Angular `dist/README.md` is too thin to teach subpath imports or the `-tw` variant (the README has been rewritten; keep it in sync)
4. `scripts/publish.mjs` allow-list drops fields silently — `files` is now in the allow-list; new package fields must also be added there to reach the installed consumer
5. Angular drift between `tsconfig.json` `paths` and `projects/<name>/` directories — `build-all.mjs` skips mismatches silently. (Historical phantom `paths` entries such as `alert-dialog`, `card`, `carousel`, `command`, `hover-card`, `pagination`, `progress`, `scroll-area`, `skeleton`, `slider`, `table`, `toggle`, `toggle-group`, `typography`, etc. have been removed; ensure new component additions update `tsconfig.json` `paths`, `angular.json` projects, AND the `projects/<name>/` directory.)
6. `scripts/publish.mjs` now removes `dist/exports.json` after copying its contents into `dist/package.json`'s `exports` field; verify that behavior stays intact so the internal build artifact does not leak to consumers
7. `prepend-use-client.mjs` globs `dist/**/*.cjs` and `dist/**/*.js` but the actual CJS output is `.js` (no `.cjs` files exist), so the `.cjs` glob is dead. If tsup's CJS extension ever changes to `.cjs` (e.g. via `outExtensions`), the `.js` glob would stop matching and CJS would lose the `"use client";` directive. Conversely `.mjs` (ESM) is never post-processed — it relies entirely on the in-tsup `prependUseClient` source plugin.

## Editing Guidance

Prefer the smallest changes that improve installed-consumer discoverability.

- to ship a new `package.json` field, also extend the `scripts/publish.mjs` allow-list
- for Angular, keep `angular.json`, `tsconfig.json` `paths`, and `projects/<name>/` in lockstep when adding/removing a component
- for React, fix the deep re-exports and the optional `exports` map together; do not document a subpath that `dist/` does not actually contain
- keep `tsup.config.ts` `external` and the source `peerDependencies` in sync
- keep `prepend-use-client.mjs`'s glob matching the tsup output formats
- improve emitted declaration usefulness before adding more prose or AI-specific files
- fix shipped README install and import examples before adding `llms.txt`
- ensure JSDoc lands on generated declarations, not only in source
- preserve `sideEffects: false`; do not drop it without verifying RSC behavior
- keep the README self-sufficient for the basic usage path

Do not invent new entrypoints, deep-import paths, sub-libraries, or future package structure just to make docs sound complete.
